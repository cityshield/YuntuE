# WebRTC P2P 文件传输测试

这是一个简单的 WebRTC 点对点文件传输测试环境，用于验证技术方案的可行性。

## 技术栈

- **WebRTC**: 点对点通信协议
- **simple-peer**: WebRTC 封装库
- **WebSocket**: 信令服务器（协调 P2P 连接）

## 文件说明

- `signaling-server.js` - WebSocket 信令服务器
- `sender.html` - 文件发送方页面
- `receiver.html` - 文件接收方页面

## 测试步骤

### 1. 启动信令服务器

```bash
cd test-p2p
node signaling-server.js
```

信令服务器将在 `ws://localhost:9000` 启动。

### 2. 打开发送方页面

在浏览器中打开 `sender.html`:

```bash
open sender.html
# 或在浏览器中直接打开文件
```

发送方页面会自动生成一个 **6 位数字配对码**（例如：123456）。

### 3. 打开接收方页面

在另一个浏览器窗口/标签页中打开 `receiver.html`:

```bash
open receiver.html
```

在接收方页面输入发送方显示的 **6 位配对码**，然后点击"连接发送方"。

### 4. 发送文件

- 在发送方页面，连接建立后会显示 "P2P 连接已建立"
- 点击"点击选择文件"按钮，选择要发送的文件
- 点击"发送文件"按钮
- 文件将通过 P2P 连接直接传输给接收方

### 5. 接收文件

- 接收方会实时显示接收进度
- 接收完成后，点击"💾 下载文件"按钮保存文件

## 工作原理

### 1. 信令交换流程

```
发送方                信令服务器              接收方
  |                      |                      |
  |--注册(sender)------->|                      |
  |                      |<-----注册(receiver)--|
  |                      |                      |
  |<----双方就绪---------|------双方就绪------->|
  |                      |                      |
  |--WebRTC Offer------->|--转发Offer---------->|
  |<---WebRTC Answer-----|<-----Answer---------|
  |                      |                      |
  |--ICE Candidate------>|--转发ICE------------>|
  |<---ICE Candidate-----|<-----ICE-------------|
  |                      |                      |
  |        P2P 连接建立（不再需要服务器）        |
  |<------------------------------------------->|
  |              直接传输文件数据                |
```

### 2. 文件传输流程

```
发送方                                        接收方
  |                                              |
  |----元数据(文件名、大小、类型)-------------->|
  |----数据块 1 (64KB)------------------------->|
  |----数据块 2 (64KB)------------------------->|
  |----数据块 3 (64KB)------------------------->|
  |    ...                                      |
  |----数据块 N (最后一块)--------------------->|
  |----结束标记(end)--------------------------->|
  |                                              |
  |                                    合并数据块 & 生成下载链接
```

### 3. 配对码系统

- **生成**: 发送方随机生成 6 位数字（100000-999999）
- **用途**: 用于在信令服务器上创建"房间"，匹配发送方和接收方
- **安全性**: 仅用于测试，生产环境需要增强（如加密、有效期、一次性等）

## 已实现的优化

### 1. 自适应高速传输 ⭐

实现了智能的自适应传输算法，可以根据网络状况自动调整传输参数：

**核心机制**:
- **动态参数调整**: CHUNK_SIZE、HIGH_WATER_MARK、SEND_DELAY 三个参数可在运行时动态调整
- **自动提速**: 连续成功传输 10 次且间隔超过 1 秒后，自动提升传输参数
  - 数据块大小增加 1.5 倍（最大 256KB）
  - 缓冲区阈值增加 1.5 倍（最大 16MB）
  - 发送延迟减少 1ms（最小 0ms）
- **自动降速**: 遇到错误时立即降低参数，确保传输稳定性
  - 数据块大小减半（最小 16KB）
  - 缓冲区阈值减半（最小 1MB）
  - 发送延迟增加 5ms（最大 20ms）

**优势**:
- 自动适应不同网络环境（WiFi、有线、跨网络）
- 稳定性优先，速度其次
- 实时显示当前传输参数
- 无需人工调优

**实现位置**: `sender.html:385-545`

### 2. 双重流控机制

实现了双重缓冲区检查，防止发送队列溢出：

**检查点**:
1. 读取文件前检查 bufferedAmount
2. 发送数据前再次检查 bufferedAmount

**重试机制**:
- 缓冲区满时等待 50ms 后重试
- 最多重试 3 次
- 失败后触发参数降级

**实现位置**: `sender.html:445-485`

### 3. 数据类型兼容

解决了 simple-peer 可能发送不同数据类型的问题：

**支持的数据类型**:
- JSON 消息（元数据、结束标记）
- ArrayBuffer
- Buffer
- Uint8Array

**检测策略**:
1. 先尝试 JSON 解析
2. 解析失败则视为二进制数据
3. 自动转换为 ArrayBuffer

**实现位置**: `receiver.html:385-463`

## 常见问题

### Q1: 信令服务器连接失败
**A**: 检查信令服务器是否已启动，端口 9000 是否被占用。

### Q2: P2P 连接无法建立
**A**: 可能是防火墙或 NAT 导致，需要配置 STUN/TURN 服务器。

### Q3: 文件传输速度慢
**A**:
- 检查网络连接质量
- 局域网内传输速度应该很快（50-100 MB/s）
- 跨网络传输受带宽限制

### Q4: 浏览器兼容性
**A**:
- Chrome: ✅ 完全支持
- Firefox: ✅ 完全支持
- Safari: ✅ 支持（需要 11+）
- Edge: ✅ 支持（Chromium 版本）
